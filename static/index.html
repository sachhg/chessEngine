Chapter 1

The Problem

I started this project with a simple observation: the best chess engines in the world play the same way against everyone. Stockfish doesn't care if it's facing a grandmaster who loves quiet positional games or a club player who throws pieces at your king. It always plays the objectively best move.

But humans are not objective. We have habits, tendencies, blind spots. A player who always grabs material will fall for a poisoned pawn. A player who avoids complications will crumble under sustained pressure. A player who loves sharp tactical battles will struggle in a dull endgame with no targets.

What if an engine could learn those patterns and turn them against you?

That's the question behind PDSO. Instead of playing the single "best" move in every position, the engine profiles its opponent's style and picks moves that specifically exploit their weaknesses. Against an aggressive player, it plays solid and lets them overextend. Against a passive player, it cranks up the pressure. Against someone who grabs every pawn, it offers poisoned material.

Before I could build that, though, I needed to understand how chess engines actually work — from the ground up.

Chapter 2

How Chess Engines Think

At the most basic level, a chess engine needs to answer one question: how good is this position? The simplest approach is material counting. Every piece has a standard value, measured in "centipawns" — hundredths of a pawn:

A pawn is worth 100 centipawns. A knight is 320, a bishop 330, a rook 500, a queen 900. The king has no value because losing it means losing the game. Add up your pieces, subtract your opponent's, and you have a rough evaluation: +300 means you're up about a minor piece.

But material isn't everything. A rook trapped in the corner is worth less than one controlling an open file. A pawn one square from promotion is nearly a queen. Position matters — a lot.

Modern engines evaluate hundreds of factors beyond material: king safety, pawn structure, piece mobility, control of key squares. In my project, I use a position complexity score that combines three sub-metrics:

complexity.py : 73-88 python 
def compute_complexity(engine, board, depth=12):
    """
    Position complexity score in [0, 1].
    Higher = harder for humans to navigate correctly.
    """
    fen = board.fen()
    if fen in _cache:
        return _cache[fen]

    spread = _eval_spread(engine, board, depth)
    tension = _tactical_tension(board)
    imbalance = _material_imbalance(board)

    score = 0.5 * spread + 0.3 * tension + 0.2 * imbalance
    _cache[fen] = round(score, 4)
    return _cache[fen]

The eval spread checks how many "reasonable" moves exist — narrow spread means a hard decision. Tactical tension counts hanging pieces and pins. Material imbalance measures how far we are from the starting position. This complexity score becomes important later when deciding how to play against different opponent types.

Chapter 3

Minimax: Looking Ahead

Knowing how to evaluate a position is only half the battle. Chess engines don't just evaluate the current position — they look ahead. Way ahead. The algorithm that makes this possible is called minimax.

The idea is intuitive: I want to maximize my evaluation, and my opponent wants to minimize it. So at each level of the game tree, we alternate between picking the move with the highest score (my turn) and the move with the lowest score (opponent's turn).

The problem? Branching factor. In an average chess position there are about 35 legal moves. At depth 1, we evaluate 35 positions. At depth 2, that's 35 × 35 = 1,225. At depth 6, it's 356 ≈ 1.8 billion positions. My laptop would melt.

Minimax Tree Visualization

Adjust the depth and watch minimax propagate values from the leaves up to the root. Green nodes maximize, red nodes minimize.

The pseudocode is deceptively simple:

pseudocode algorithm 
def minimax(node, depth, maximizing):
    if depth == 0 or node.is_terminal():
        return evaluate(node)

    if maximizing:
        best = -infinity
        for child in node.children:
            val = minimax(child, depth - 1, False)
            best = max(best, val)
        return best
    else:
        best = +infinity
        for child in node.children:
            val = minimax(child, depth - 1, True)
            best = min(best, val)
        return best
Chapter 4

Alpha-Beta Pruning

Minimax works, but it's way too slow for real chess. Enter alpha-beta pruning — the single most important optimization in game-tree search.

The core insight: if I already know I can guarantee a score of +2 from a previous branch, and I start exploring a new branch where my opponent can force a score of +1, I can stop immediately. That branch is already worse than what I have. Why prove it's even worse?

Alpha is the best score the maximizing player can guarantee. Beta is the best score the minimizing player can guarantee. Whenever alpha ≥ beta, we prune — cut off the branch. In practice, this reduces the effective branching factor from ~35 to ~6, turning an impossible search into a tractable one.

Alpha-Beta Pruning

Watch the same tree searched with alpha-beta. Greyed-out branches were pruned — never explored. Compare the node count to pure minimax.

With good move ordering (trying the most promising moves first), alpha-beta can prune over 90% of the tree. Stockfish uses this as its foundation, layered with dozens of additional optimizations — null move pruning, late move reductions, aspiration windows — but alpha-beta is the backbone.

Chapter 5

NNUE: The Neural Revolution

For decades, chess engines used handcrafted evaluation functions — hundreds of if-then rules coded by programmers. Piece-square tables, king safety heuristics, pawn structure bonuses.

Then in 2020, everything changed. Stockfish adopted NNUE (Efficiently Updatable Neural Network), and its playing strength jumped by roughly 80 Elo points overnight.

NNUE is a small neural network — just a few megabytes — that replaces the entire handcrafted evaluation function. It takes the board state as input (which piece is on which square) and outputs a single number: the evaluation. The key insight: the network can be incrementally updated as pieces move, making it fast enough for millions of evaluations per second.

NNUE made engines incredibly strong at finding the objectively best move. But that raises an interesting question: best for what? Best against a perfect opponent? Or best against the specific, imperfect human sitting across the board?

That question is the foundation of my project.

Chapter 6

Opponent Modeling

This is where my project begins. The core idea: if I can profile how a specific opponent plays — their tendencies, their habits, their weaknesses — I can tune the engine to exploit those patterns.

I started by asking: what measurable features distinguish one player's style from another? After studying games from players with very different styles, I settled on eight behavioral features, each extracted from a player's game history:

Chapter 7

The Five Archetypes

With these features in hand, I needed a way to categorize players into distinct styles. I settled on five archetypes, each inspired by legendary chess players whose games epitomize that style.

Archetype Gallery

Click an archetype to see its behavioral signature — the centroid values used for classification.

Classification is straightforward: compute the Euclidean distance from a player's feature vector to each archetype's centroid, and assign them to the nearest one. No machine learning, no black boxes — just geometry in eight dimensions.

profiler.py : 254-264 python 
def classify(features: dict) -> dict:
    """Classify feature vector into nearest archetype by Euclidean distance."""
    distances = {}
    for arch, centroid in CENTROIDS.items():
        dist_sq = 0.0
        for feat, val in centroid.items():
            player_val = features.get(feat, 0.5)
            dist_sq += (player_val - val) ** 2
        distances[arch] = round(math.sqrt(dist_sq), 4)

    best = min(distances, key=distances.get)

Chapter 8

Counter-Strategy Design

Now comes the heart of the project: once I know what kind of player I'm facing, how do I exploit their tendencies?

The answer lies in MultiPV (multiple principal variations). Instead of asking Stockfish for just the single best move, I ask for the top 3. Then I pick the one that best counters the opponent's style — as long as it's within a tolerable eval range of the top move.

Here's the full move selection logic — the core of the adaptive engine:

engine.py : 161-227 python 
def _select_counter_move(self, board, info_list):
    """Among MultiPV candidates, pick the one that best exploits the opponent."""
    if len(info_list) == 1:
        return info_list[0]["pv"][0]

    behavior = COUNTER_BEHAVIOR.get(self.archetype, {})
    tolerance = behavior.get("sacrifice_tolerance", 50)

    top_score = info_list[0]["score"].pov(board.turn)
    top_cp = top_score.score(mate_score=10000)

    # If already clearly winning, just play the best move
    if top_cp > 200:
        return info_list[0]["pv"][0]

    if self.archetype == Archetype.MATERIALISTIC:
        # Offer a sacrifice: material goes down but eval stays close
        for entry in info_list[1:]:
            mv = entry["pv"][0]
            cp = entry["score"].pov(board.turn).score(mate_score=10000)
            if abs(top_cp - cp) < tolerance:
                mat_before = _material_on_board(board)
                sim = board.copy()
                sim.push(mv)
                if mat_before - _material_on_board(sim) > 50:
                    return mv

    elif self.archetype == Archetype.PASSIVE:
        # Open the position — captures, pawn breaks
        for entry in info_list:
            mv = entry["pv"][0]
            cp = entry["score"].pov(board.turn).score(mate_score=10000)
            if abs(top_cp - cp) < tolerance and board.is_capture(mv):
                return mv

    elif self.archetype == Archetype.TACTICAL:
        # Simplify: force trades
        for entry in info_list:
            mv = entry["pv"][0]
            cp = entry["score"].pov(board.turn).score(mate_score=10000)
            if abs(top_cp - cp) < tolerance:
                sim = board.copy()
                sim.push(mv)
                if _material_on_board(sim) < _material_on_board(board) - 200:
                    return mv

    elif self.archetype == Archetype.AGGRESSIVE:
        # Solid, defensive positions
        quiet = [e for e in info_list
                 if _is_quiet_move(board, e["pv"][0])
                 and abs(top_cp - e["score"].pov(board.turn)
                         .score(mate_score=10000)) < tolerance]
        if quiet:
            return quiet[0]["pv"][0]

    elif self.archetype == Archetype.POSITIONAL:
        # Inject imbalances
        for entry in info_list:
            mv = entry["pv"][0]
            cp = entry["score"].pov(board.turn).score(mate_score=10000)
            agg = _aggression_score(board, mv)
            if abs(top_cp - cp) < tolerance and agg > 1.0:
                return mv

    return info_list[0]["pv"][0]

  2. Baseline mode — default Stockfish with no adaptation

@property
def elo_diff(self) -> float:
    """Estimate Elo difference from score."""
    s = self.score
    if s <= 0.0:
        return -400.0
    if s >= 1.0:
        return 400.0
    return -400.0 * math.log10(1.0 / s - 1.0)

What I Learned

Building this taught me that the "best" move in chess depends on context. Against a perfect opponent, the objectively best move is always correct. But against a human with known tendencies, a slightly suboptimal move that exploits their weakness can be more effective.

The gap between "theoretically best" and "practically best" is where adaptation lives. And that gap is measurable.
